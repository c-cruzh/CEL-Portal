# Auditoría de implementación vs Macro PRD (2026-05-19)

## Alcance
Revisión estática del código actual (`artifacts/portal` + `artifacts/api-server`) para identificar qué partes del macro PRD ya están implementadas, parcialmente implementadas o pendientes.

## Resumen ejecutivo

- **Configuración / Admin portal**: **Parcialmente implementado**. Existe un portal admin con tabs para miembros, invitaciones, roles, notificaciones y bitácora; aún requiere UX simplification y validación humana para varios mapeos.  
- **Equipo (roles, perfiles, asignaciones, invitaciones)**: **Mayormente implementado** para CRUD operativo básico y flujos de invitación; faltan ajustes de lógica fina y reglas de negocio más estrictas del PRD.  
- **Cronograma**: **Parcialmente implementado**. Hay fases/entregables con drill-down visual, pero la trazabilidad completa al documento OG y granularidad total requiere validación adicional.
- **Calendario**: **Parcialmente implementado**. Existe lógica de weekly autogenerada e importación batch de sesiones, pero se reportó regresión del enlace con T0 y necesita QA funcional extremo.
- **Metodología**: **Parcialmente implementado**. Secciones estructuradas, detalle por fase presente (y debe congelarse), pero quedan mejoras de visualización/animación y reestructuración de infraestructura.
- **Desarrollo técnico**: **Parcialmente implementado** con avances claros en decisiones/dependencias; múltiples capítulos siguen requiriendo rediseño y curaduría manual desde OG.
- **Kanban**: **Parcialmente implementado** con batch import y categorías, pendiente consolidación operativa y curaduría de tareas preproyecto vs piloto.
- **Decisiones**: **Implementado en base** (incluye batch import y campos de ciclo de vida), pero pendiente robustecer taxonomía y gobernanza exacta solicitada.

## Hallazgos por módulo

### 1) Módulo EQUIPO

**Implementado**
- Gestión administrativa centralizada en Configuración con tabs de miembros, invitaciones y roles.  
- Gestión de invitaciones admin (`listar`, `crear`, `reenviar`, `revocar`) en backend.  
- Edición de miembros y asignación de múltiples roles por persona.  
- Señales explícitas de validación humana para mapeo de roles split (Kevin).  
- Soporte de CV del miembro en UI administrativa (visualización/estado y eliminación al guardar según flujo).  

**Parcial/Pendiente**
- Restricción “admins solo Camila y Kevin” existe de forma semiautomática en auth bootstrap, pero se recomienda endurecer en política explícita configurable + pruebas de autorización end-to-end.
- Reglas exactas de “roles requeridos” del PRD todavía necesitan cierre funcional y validación manual.
- Campos de contacto (teléfono/correo) están parcialmente cubiertos por correo en flujos de invitación y búsqueda; validar explícitamente teléfono obligatorio por rol si eso era requisito duro.

### 2) Módulo CRONOGRAMA

**Implementado**
- Vista de cronograma mensual con fases y actividades.
- Estructura con anclas por fase (`fase-*`) y relación con otras vistas.
- Componente de involucramiento por fase/roles y enlaces cruzados desde metodología.

**Parcial/Pendiente**
- Confirmar que el drill-down actual cubre toda la granularidad del OG DSP (no solo subset).
- Validar que cada fase tenga población exhaustiva de individuos/roles reales (no placeholders), y que sincronice con equipo.

### 3) Módulo CALENDARIO

**Implementado**
- Soporte de batch import de sesiones (dialog dedicado).
- Lógica server-side de sesiones semanales (`weeklies`) con modo idempotente.
- Exportación iCal se mantiene en librería backend.

**Parcial/Pendiente crítico**
- El PRD reporta rotura del enlace T0 → autocreación weeklys; hay utilidades para weeklys, pero se requiere verificación funcional en runtime para confirmar que el trigger sigue activo tras cambios recientes.
- Validación humana previa al batch import está parcialmente implícita en copy/flujo, pero falta enforcement formal por estados o workflow de aprobación.

### 4) Módulo METODOLOGÍA

**Implementado**
- Estructura de navegación por grupos (General/Infraestructura/Ruta/Detalle por fase/Operación).
- Sección “Detalle por fase” claramente implementada y conectada con cronograma.
- Señales de contenido “Por validar (Kevin)”.

**Parcial/Pendiente**
- Pendiente mayor: hacer la vista “3 phase HL” más visual/animada y menos textual.
- Infraestructura aún requiere refactor para separar claramente categorías (infra física/cómputo vs F1/F2/F3 subtipos), según PRD.
- “Ruta y entregables” requiere de-duplicación respecto cronograma y más granularidad de colaboración.

### 5) Módulo DESARROLLO TÉCNICO

**Implementado**
- Capítulo de decisiones abiertas/dependencias con CEL está presente y navegable.
- Capítulo RACI y detalle de tareas por fase está integrado.
- Señales explícitas de validación humana en ciertos bloques.

**Parcial/Pendiente**
- Reestructuración de ETL y chapters 9/10/11 sigue incompleta respecto al nivel de detalle del documento OG.
- Secciones declaradas como “freeze” en PRD (modelos de predicción y sección 5) deben auditarse con test de no-regresión de contenido (actualmente no automatizado).

### 6) KANBAN

**Implementado**
- Batch import para tarjetas Kanban (JSON/CSV + plantilla + preview).
- Backend/admin routing para operaciones Kanban.

**Parcial/Pendiente**
- Definición final de categorías preproyecto vs piloto sigue siendo una decisión de negocio pendiente (requiere input de Kevin).
- Falta evidencia de QA extensivo automatizado y cobertura de lógica límite.

### 7) DECISIONES

**Implementado**
- CRUD de decisiones, estados, resolver/reabrir, ownership y fechas.
- Batch import de decisiones con estructura de opciones y validación básica.

**Parcial/Pendiente**
- PRD pide ampliar ciclo de vida + registrar qué decisión se tomó, quién y cuándo: hay base funcional, pero conviene revisar que todos los campos obligatorios del flujo real queden enforced y visibles en UI/resúmenes.

### 8) CONFIGURACIÓN (UX Admin)

**Implementado**
- Existe “admin portal” con funcionalidades reales (miembros, invitaciones, roles, notificaciones, bitácora).

**Parcial/Pendiente**
- Persisten problemas de usabilidad reportados (“counterintuitive”). Requiere rediseño de navegación, textos de ayuda y reducción de fricción en tareas frecuentes (invitar, asignar roles, batch ops).

## Matriz rápida “Hecho vs No Hecho”

- **Hecho / Base funcional presente**: admin tabs, invitaciones, roles catalog, multi-role por usuario, CV flow, batch import (kanban/decisiones/sesiones), decisiones lifecycle básico, iCal export, cronograma por fases.
- **No cerrado / requiere validación humana**: reglas finales de roles requeridos, mapeo OG vs versión final (Kevin), taxonomía kanban preproyecto/piloto, curaduría de capítulos técnicos deprecados, validación de T0 weekly linkage en entorno real.
- **Pendiente UX fuerte**: “pimp up admin portal”, simplificación de configuración y guías de uso.
