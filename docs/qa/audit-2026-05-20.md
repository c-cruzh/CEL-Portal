## Verificación E2E (paso 5)

Suite Playwright propia en `tests/e2e/` (`@workspace/e2e`), ejecutable vía
el workflow registrado **`tests/e2e: Playwright`** o `pnpm --filter
@workspace/e2e test`.

**Hardening del bypass de auth** (`lib/e2eTestMode.ts`). La cuenta Clerk
del piloto es una instancia *development* sin Testing Tokens, así que el
API expone un bypass de auth para que las pruebas no consuman la cuenta
real. El bypass se considera código sensible y por eso exige **las
cuatro** condiciones a la vez antes de saltarse Clerk:

1. `NODE_ENV !== "production"` (verificado en `isE2ETestModeEnabled`).
2. `process.env.E2E_TEST_MODE === "1"` (sólo lo setea el script `dev`
   de `artifacts/api-server`; el bloque `[services.production]` no lo
   expone, así que es imposible activarlo en producción).
3. La petición entró por loopback (`req.ip` ∈ `{127.0.0.1, ::1,
   ::ffff:127.0.0.1}`), bloqueando llamadas desde fuera del contenedor.
4. Header `x-e2e-secret` que matchea exactamente, con `timingSafeEqual`,
   el secreto generado en el arranque del api-server. El secreto se
   guarda en `/tmp/portal-cel-e2e-bypass-secret` con permisos `0600` —
   **no está en git, no está en variables de entorno, y nunca se hace
   eco en una respuesta HTTP**. Sólo procesos en el mismo contenedor
   pueden leerlo.

Las pruebas leen el secreto del filesystem compartido (`tests/e2e/tests/
helpers.ts` → `getBypassSecret`) y lo inyectan en cada request. Se
incluyen 3 tests de regresión para garantizar que el bypass falla
cerrado:
- sin el header `x-e2e-user-email` (sólo `x-e2e-admin`) → 401.
- con `x-e2e-user-email` + `x-e2e-admin` pero **secreto falso** → 401.
- con `x-e2e-user-email` + `x-e2e-admin` pero **sin secreto** → 401.
